# LAD CLPD Backend

REST API for the Dubai Legal Affairs Department's CLPD (Continuing Legal Professional Development) platform.

- **Stack:** Node.js 20+ · Express · SQLite (better-sqlite3) · JWT
- **Auth:** UAE Pass OAuth 2.0 / OpenID Connect (with email + password fallback for staff)
- **Data:** Seeded from `data/Blank_data_25.xlsx` (the LAD report)

---

## Quick start

```bash
# 1. Install dependencies
npm install

# 2. Configure environment
cp .env.example .env
# …edit .env, see below

# 3. Initialise the database
npm run init-db

# 4. Seed from the LAD spreadsheet
#    (Make sure data/Blank_data_25.xlsx is present)
npm run seed

# 5. Start
npm start            # production
npm run dev          # with auto-reload

# Verify
curl http://localhost:4000/api/v1/health
```

Or with Docker:

```bash
docker-compose up
```

This launches the API on port `4000` and a static nginx serving the frontend on `8080`.

## UAE Pass integration

UAE Pass (https://uaepass.ae) is the UAE government's national digital identity, supporting OAuth 2.0 / OpenID Connect. This backend implements the full flow.

### How it works

1. **User clicks "Login with UAE Pass"** in the frontend → browser opens `GET /api/v1/auth/uaepass/login`.
2. Backend generates a CSRF `state` token and a PKCE `code_verifier`, stores both in `oauth_state`, and redirects the user to UAE Pass's `/idshub/authorize` endpoint with a `code_challenge`.
3. UAE Pass authenticates the user (Emirates ID + biometric / OTP), then redirects the browser back to `GET /api/v1/auth/uaepass/callback?code=…&state=…`.
4. Backend looks up the `state`, exchanges the `code` for an `access_token` at `/idshub/token` (Basic auth, PKCE verifier), and fetches the user profile from `/idshub/userinfo`.
5. Backend maps the UAE Pass profile to a local user:
   - by `uaepass_uuid` (returning user)
   - or by Emirates ID (`idn` field — first-time login, automatically links accounts)
   - or by email (back-office staff)
6. Issues a JWT and redirects the browser to the frontend with the token in the URL fragment: `…#token=…&role=…&name=…`.
7. The frontend stores the token in `localStorage` and includes `Authorization: Bearer …` on every API call.

### Endpoints used

| Endpoint | Staging | Production |
|---|---|---|
| `/authorize` | `https://stg-id.uaepass.ae/idshub/authorize` | `https://id.uaepass.ae/idshub/authorize` |
| `/token` | `https://stg-id.uaepass.ae/idshub/token` | `https://id.uaepass.ae/idshub/token` |
| `/userinfo` | `https://stg-id.uaepass.ae/idshub/userinfo` | `https://id.uaepass.ae/idshub/userinfo` |
| `/logout` | `https://stg-id.uaepass.ae/idshub/logout` | `https://id.uaepass.ae/idshub/logout` |

The base URL is selected by `UAEPASS_ENV` (`staging` or `production`).

### Profile fields received

From `/idshub/userinfo` you get (subject to scope and `userType`):

```json
{
  "uuid":          "13b91a…",            // UAE Pass internal ID
  "unifiedID":     "9b21…",              // cross-platform stable ID
  "idn":           "784-1985-1234567-1", // Emirates ID
  "firstnameEN":   "Yousef",  "firstnameAR": "يوسف",
  "lastnameEN":    "Al Mansouri", "lastnameAR": "المنصوري",
  "email":         "y.mansouri@…",
  "mobile":        "+9715…",
  "nationalityEN": "United Arab Emirates",
  "gender":        "Male",
  "dob":           "1985-04-12",
  "userType":      "SOP3"                // SOP1=visitor, SOP2=email/mobile verified, SOP3=fully verified
}
```

`userType` is important: SOP1 is a self-registered visitor with no Emirates ID, SOP3 is fully verified at a UAE Pass kiosk. For accessing lawyer records, you almost certainly want to require `SOP2` or `SOP3`. Add the check in `src/routes/auth.js` if needed.

### Onboarding with UAE Pass

To go live, you need to register your service provider:

1. Email `support@uaepass.ae` requesting SP onboarding.
2. Provide your domain, redirect URI, and use case.
3. They issue you `client_id` and `client_secret` for **staging first**, then **production** once you pass review.
4. Set `acr_values` to `urn:safelayer:tws:policies:authentication:level:low` initially. Once approved, you may use `…level:high` for sensitive flows like digital signing.

**Test credentials for staging**: `sandbox_web_stage` / (request secret from UAE Pass). Test users are available in the UAE Pass mobile app's staging environment — see https://docs.uaepass.ae for the list of test Emirates IDs.

### Common pitfalls

- **`redirect_uri` mismatch.** The redirect URI you register with UAE Pass must match `UAEPASS_REDIRECT_URI` *exactly* — including scheme, host, port, and path.
- **`Content-Type` on `/token`.** Use `application/x-www-form-urlencoded`. (UAE Pass's own docs once suggested `multipart/form-data`; the spec-compliant form works in both environments.)
- **PKCE.** Even though our client is confidential, we use PKCE for defence in depth. UAE Pass accepts it without issue.
- **Staging is UAE-only.** UAE Pass staging endpoints are firewalled to UAE IP ranges. If developing remotely, use a UAE VPN or a remote box.

## API reference

All endpoints under `/api/v1/`. JSON in, JSON out. Auth via `Authorization: Bearer <jwt>`.

### Auth

| Method | Path | Auth | Description |
|---|---|---|---|
| GET  | `/auth/uaepass/login` | — | Start UAE Pass OAuth (redirects to UAE Pass) |
| GET  | `/auth/uaepass/callback` | — | UAE Pass redirects back here (don't call directly) |
| GET  | `/auth/uaepass/logout` | Bearer | Single sign-out via UAE Pass |
| GET  | `/auth/me` | Bearer | Current user profile |
| POST | `/auth/logout` | Bearer | Local logout — revoke JWT |
| POST | `/auth/staff/login` | — | Email + password (LAD admin / firm CO fallback) |

### Courses & content

| Method | Path | Auth | Description |
|---|---|---|---|
| GET    | `/courses` | optional | List active courses |
| GET    | `/courses/:id` | optional | One course with its sessions |
| POST   | `/courses` | `lad_admin`, `provider_admin` | Create course |
| PUT    | `/courses/:id` | `lad_admin`, `provider_admin` | Upsert course |
| DELETE | `/courses/:id` | `lad_admin` | Delete course |
| GET    | `/courses/sessions/all` | optional | All upcoming sessions |
| POST   | `/courses/sessions/bulk` | `lad_admin`, `provider_admin` | Bulk upsert sessions |
| GET    | `/content` | — | CMS strings (homepage hero, KPIs, etc.) |
| PUT    | `/content` | `lad_admin` | Update CMS strings |
| GET    | `/faq` | — | FAQ list |
| PUT    | `/faq` | `lad_admin` | Replace FAQ list |

### People

| Method | Path | Auth | Description |
|---|---|---|---|
| GET | `/lawyers/me` | Bearer (lawyer) | Current lawyer's profile + bookings |
| GET | `/lawyers/:id` | Bearer | Specific lawyer (self / firm CO / LAD admin) |
| GET | `/firms` | `lad_admin`, `lad_intelligence` | All firms |
| GET | `/firms/:id` | Bearer | Firm details |
| GET | `/firms/:id/lawyers` | Bearer (LAD or own CO) | Lawyers in a firm |
| GET | `/firms/:id/bookings` | Bearer (LAD or own CO) | Recent bookings |

### Bookings

| Method | Path | Auth | Description |
|---|---|---|---|
| POST   | `/bookings` | Bearer | Create booking |
| PATCH  | `/bookings/:id` | Bearer | Update status / points / notes |

### Stats

| Method | Path | Auth | Description |
|---|---|---|---|
| GET | `/stats/aggregate` | — | Total practitioners / firms / compliance % |
| GET | `/stats/firms` | LAD | Per-firm compliance for current year |
| GET | `/stats/firm/:id` | Bearer | Compliance breakdown for one firm |

### Composite (used by the public portals)

| Method | Path | Description |

## Roles

The `role` claim in the JWT controls authorisation. Defined roles:

| Role | Where they log in | Sees |
|---|---|---|
| `lawyer` | Any practising lawyer (via UAE Pass) | Own profile, own bookings, public catalog |
| `firm_compliance_officer` | Firm CO (UAE Pass or password) | All lawyers + bookings in their firm |
| `lad_admin` | LAD course administrator | Everything; CMS write |
| `lad_intelligence` | LAD analyst (read-only) | Cross-firm dashboards, no writes |
| `provider_admin` | Accredited provider rep | Their courses + attendance filing |

## Database schema

See `src/schema.sql` for the full DDL. Core tables:

- **firms** — 499 practising firms from the LAD report
- **providers** — 11 accredited course providers
- **courses** — 600+ unique course titles (active + historical)
- **course_sessions** — scheduled sittings (date, venue, capacity, seats remaining)
- **lawyers** — 6,520 lawyer records (4,411 practising)
- **staff** — LAD admins + firm COs (separate from lawyers)
- **bookings** — 154,352 booking records spanning 2015–2026
- **credit_transactions** — credit purchases & refunds
- **auth_sessions** — JWT issuance log (for revocation)
- **oauth_state** — short-lived CSRF tokens for UAE Pass callbacks
- **audit_log** — every login + privileged action
- **content** + **faq** — CMS strings

Migration to PostgreSQL is straightforward — better-sqlite3 prepared statements use the same `?` placeholder convention. Swap `db.js` for a `pg` Pool and the rest of the code is unchanged.

## Frontend wiring

In the frontend (the HTML portals), point `window.LAD_API_BASE` at this backend before any portal HTML runs:

```html
<script>
  window.LAD_API_BASE = 'https://api.your-domain.ae';  // or http://localhost:4000 for dev
</script>
```

The `api-client.js` helper (in the frontend) then routes all calls through this base URL. After UAE Pass login, the user lands on the frontend with `#token=…&role=…&name=…` in the URL — the frontend's auth-bridge stores this in `localStorage.lad_token` and includes it as `Authorization: Bearer …` on every subsequent fetch.

## Deployment notes

### Azure App Service (recommended for UAE government)

1. Provision an Azure App Service Plan (B1 or higher) in `uaenorth` (Abu Dhabi) for low latency to UAE Pass.
2. Set the app settings (from `.env`) under Configuration → Application Settings.
3. Deploy via GitHub Actions, ZIP deploy, or `az webapp up`.
4. Use Azure SQL or Azure Database for PostgreSQL Flexible Server instead of SQLite for production.
5. Configure Application Insights for monitoring.

### Secrets

Never commit:
- `.env`
- `data/lad-clpd.sqlite` (PII)
- `data/Blank_data_25.xlsx` (PII)
- `UAEPASS_CLIENT_SECRET`
- `JWT_SECRET`
- `ANTHROPIC_API_KEY`

Add these to `.gitignore` (already done in this repo).

### CORS

Update `CORS_ORIGIN` in `.env` to a comma-separated list of allowed frontend origins. UAE government convention is `*.dubai.gov.ae`.

### Rate limiting

The default is 120 requests per minute per IP. Adjust `RATE_LIMIT_*` for higher tiers. The UAE Pass callback endpoint is excluded so re-auth retries don't fail.
